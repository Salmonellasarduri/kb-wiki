# Andrej Karpathyによる個人用知識ベース構築のLLMワークフロー

Andrej Karpathy（アンドレイ・カルパシー）が提案する、LLMを活用した個人用知識ベース構築の包括的なワークフローについて解説する。このアプローチでは、従来のコード操作よりも知識操作に重点を置き、マークダウンと画像として保存された知識を体系的に管理する。

## データ収集プロセス

### 生データの蓄積
- 論文、記事、リポジトリ、データセット、画像などのソース文書を`raw/`ディレクトリに索引化
- Obsidian Web Clipper拡張機能を使用してWeb記事を`.md`ファイルに変換
- ホットキーを活用して関連画像をローカルダウンロードし、LLMが容易に参照できる環境を構築

### LLMによるコンパイル
LLMが生データを段階的に「コンパイル」してwikiを構築する。このwikiは以下の要素で構成される：
- ディレクトリ構造化された`.md`ファイルのコレクション
- `raw/`内全データの要約
- バックリンク機能
- 概念別にカテゴライズされたデータと記事
- 相互リンクによる知識ネットワーク

## 開発環境とインターフェース

### Obsidianを活用したIDE
- Obsidianを「フロントエンド」IDEとして使用
- 生データ、コンパイル済みwiki、派生的な可視化を一元的に閲覧
- **重要な特徴**：LLMがwikiのすべてのデータを作成・維持し、人間はほとんど直接編集しない
- Marpプラグインなど、様々なObsidianプラグインでデータを多様な形式で表示

## スケーラブルなQ&Aシステム

### 大規模知識ベースへの対応
wikiが十分な規模（約100記事、400K語程度）に達すると、以下が可能になる：
- LLMエージェントに対する複雑な質問
- 自動的な関連情報の調査・回答生成
- 小規模では高度なRAG（Retrieval-Augmented Generation）は不要
- LLMによる索引ファイルと文書要約の自動維持により、関連データへの効率的なアクセス

## 多様な出力形式

### 視覚的な知識表現
回答をテキスト/ターミナル出力ではなく、以下の形式で生成：
- マークダウンファイル
- スライドショー（Marp形式）
- matplotlib画像
- その他のクエリ依存的な視覚フォーマット

### 知識の循環的強化
出力結果をwikiに「ファイリング」して戻すことで、以下を実現：
- さらなるクエリのためのwiki強化
- 個人的な探索と質問が知識ベースに累積
- 自己強化型の学習サイクル

## データ品質の維持

### LLMによる健全性チェック
以下のような「ヘルスチェック」を実行：
- 一貫性のないデータの発見
- Webサーチャーを使った欠損データの補完
- 新記事候補となる興味深いつながりの発見
- wikiの段階的なクリーンアップとデータ整合性向上
- LLMによるさらなる調査質問の提案

## 追加ツールの開発

### 専用検索エンジン
wiki専用の小規模で単純な検索エンジンを開発：
- Web UIでの直接利用
- より大きなクエリのためのCLIツールとしてLLMに提供
- ナイーブだが実用的なアプローチ

## 将来の発展方向

### 合成データ生成とファインチューニング
リポジトリの成長に伴い、以下への展開が考えられる：
- 合成データ生成
- コンテキストウィンドウではなく、LLMの重みに知識を「記憶」させるファインチューニング

## システムアーキテクチャの要約

**TLDR**：複数ソースからの生データ収集 → LLMによる`.md`wikiへのコンパイル → LLMのCLIによるQ&A実行とwiki段階的強化 → すべてをObsidianで閲覧。人間によるwikiの手動編集・作成はほとんど行わず、LLMの領域として運用。

現在はスクリプトの寄せ集めとして実装されているが、将来的には統合された製品としての大きな可能性を秘めている。
