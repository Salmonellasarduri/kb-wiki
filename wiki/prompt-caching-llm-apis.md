# LLM API のプロンプトキャッシュ機能

プロンプトキャッシュは、LLM APIプロバイダーが以前に処理したプロンプトを保存し、同じプレフィックスを持つ後続のリクエストでレスポンス時間とコストを削減する技術である。

## 動作メカニズム

1. APIプロバイダーがプロンプトプレフィックスをハッシュ化
2. キャッシュされたバージョンが存在する場合、モデルはそれらのトークンの再処理をスキップ
3. プロンプトの新規部分・差分部分のみが処理される
4. 結果として、レスポンス時間の短縮とコスト削減（通常50-90%の節約）を実現

## 主要プロバイダー

### Anthropic
- 1024トークンを超えるプロンプトで自動キャッシュが有効
- キャッシュTTL（生存時間）は5分
- キャッシュされたトークンは通常の入力レートの10%で課金

### OpenAI
- 同様のメカニズムで自動キャッシュを提供
- キャッシュされたトークンに対して50%の割引

### Google
- Gemini APIでコンテキストキャッシュを利用可能
- 設定可能なTTLを提供

## ベストプラクティス

- **静的コンテンツの前置**: システムプロンプトや参考文書などの変更頻度の低いコンテンツをプロンプトの先頭に配置
- **動的コンテンツの後置**: ユーザーメッセージや現在のコンテキストなど変動する内容を末尾に配置
- **プレフィックスの安定化**: リクエスト間でキャッシュ可能なプレフィックス部分を一定に保つ
- **監視**: APIレスポンスヘッダーからキャッシュヒット率を監視

## ナレッジベース設計への影響

LLM APIでナレッジベースを活用する際、プロンプトキャッシュは以下の意味を持つ：

- **再利用の経済性**: 同じWiki記事を繰り返し読み込むことが初回以降は非常に安価になる
- **静的インデックスの優位性**: 変更頻度の低いインデックスファイルがキャッシュの恩恵を最も受ける
- **コンテキスト注入順序の重要性**: 安定したコンテンツを先に配置することが効率化のカギとなる
