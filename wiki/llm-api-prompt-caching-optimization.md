# LLM APIにおけるプロンプトキャッシング最適化技術

プロンプトキャッシングは、LLM APIプロバイダーが以前に処理されたプロンプトを保存し、同じプレフィックスを持つ後続のリクエストでレイテンシとコストを削減する技術である。

## 動作メカニズム

プロンプトキャッシングは以下のプロセスで動作する：

1. **プレフィックスのハッシュ化**: APIプロバイダーがプロンプトのプレフィックス部分をハッシュ化
2. **キャッシュ検索**: キャッシュされたバージョンが存在する場合、モデルはそれらのトークンの再処理をスキップ
3. **差分処理**: プロンプトの新規・差分部分のみが処理される
4. **コスト削減**: 通常50-90%のコスト削減と低遅延を実現

## 主要プロバイダーの実装

### Anthropic
- 1024トークンを超えるプロンプトで自動キャッシングが有効
- キャッシュTTL（生存時間）は5分
- キャッシュされたトークンの課金は通常の入力レートの10%

### OpenAI
- 類似のメカニズムで自動キャッシングを実装
- キャッシュされたトークンに対して50%の割引

### Google
- Gemini APIでコンテキストキャッシングが利用可能
- TTLの設定が可能

## ベストプラクティス

効果的なプロンプトキャッシング活用のための推奨事項：

- **静的コンテンツの前置**: システムプロンプトや参照ドキュメントなどの静的コンテンツをプロンプトの先頭に配置
- **動的コンテンツの後置**: ユーザーメッセージや現在のコンテキストなどの動的コンテンツを末尾に配置
- **プレフィックスの安定性**: リクエスト間でキャッシュ可能なプレフィックス部分を安定的に保つ
- **監視と最適化**: APIレスポンスヘッダーを通じてキャッシュヒット率を監視

## 知識ベース設計への影響

LLM APIでの知識ベース運用においてプロンプトキャッシングが与える影響：

- **記事再読込の効率化**: 同じwiki記事を繰り返し読み込む処理が初回後は安価になる
- **インデックスファイルの最適化**: 変更頻度の低いインデックスファイルがキャッシングの恩恵を最も受ける
- **コンテキスト注入順序の重要性**: 安定したコンテンツを先頭に配置する順序設計が重要

この技術は特に[[andrej-karpathy-llm-knowledge-base-workflow]]のような大規模な知識ベース運用において、コスト効率とパフォーマンスの大幅な改善をもたらす。
